# 任务信息结构化设计的演化（本体锁定问题）

早期任务数据只有自然语言提示词（prompt）。

所有信息：

- 约束条件
- 难度
- 风格
- 预期失败模式

全部压缩在提示词中。

起初这样做非常自然。

因为提示词是最灵活的。

但随着不良案例（bad case）增多，一个问题逐渐浮现：

系统无法显式地知道：

> “这道题到底在测什么。”

例如：

无法筛选：

- 高约束密度任务
- 长程语义压力任务
- 风格迁移任务

也无法分析：

- 哪类压力最容易导致失稳
- 预期失败模式与实际失败模式之间的偏差

所有归因都依赖人脑记忆。

于是我在设计系统时开始尝试对任务信息进行结构化。

但新的问题很快出现：

如果一开始就把所有概念固化成模式，
系统设计就会被当前的理解所锁死。

因为：

对失稳模式的理解本身仍在演化。

最终形成了分层结构：

- L0：强稳定结构
- L1：半稳定解释
- L2：开放文本

其中：

L0 包含：

- 约束条件
- 难度轴

这些字段会被代码直接消费，
因此必须长期保持稳定。

L1 包含：

- 预期失败模式

它代表当前阶段对失稳模式的理解，
允许被频繁修改。

L2 则保留：

- 潜在张力

用于容纳尚未结构化、但可能重要的解释空间。

这次演化带来的核心变化是：

系统开始主动防止：

**本体锁定**

即：

> 不让当前认知阶段的分类体系，
> 过早冻结未来的解释空间。
